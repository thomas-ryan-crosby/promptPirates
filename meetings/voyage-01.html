Tools, testing & the boring stack

A builders' roundtable on AI coding agents in practice

Date
Wed, June 10, 2026 · 6:01 PM · 1:03:30
Aboard
Thomas Ryan Crosby (CrumbLabz), Ryan Lavin (telehealth startup), Nathan Allen (ranch-management app)
Hailed
Alex Boudreaux, Josh Meister, Vincenzo Caronna
Format
Informal peer call — show-and-tell of current AI projects, tooling, and workflows. Several participants were mobile (driving, loading a car), so it ran loose and tangent-heavy.
Weigh anchor The maiden voyage set sail at six bells on a Wednesday — three captains aboard, half of them steering from the open road, one loading a car between broadsides. Charts were compared, war stories swapped, and a fair haul of lessons hauled up from the deep. What follows is the log, plain and true. Mind the treasure chest near the stern.

A recurring builders' roundtable to compare how each person is using AI coding agents in practice — what's working, what they've stopped doing, and where the tooling and industry are heading.

1Models, subscriptions & cost

  • Everyone is bumping into usage limits and drifting toward API / usage-based billing. Ryan Lavin blew through two Claude Max subscriptions' 5-hour limits in ~an hour and switched to pay-as-you-go; noted Fable goes API-only after June 22.
  • General hope that the "subsidy era" of cheap/free model access lasts as long as possible.
  • Fable (newest model) is the consensus favorite — reportedly tops the new "Deep Sweet" long-horizon coding benchmark by a wide margin. Standout traits: terminal use, adversarial thinking, and running large agent swarms.
  • Some switched from Claude to Codex purely for better usage limits + engineering performance, while acknowledging Claude is stronger at UI.

2Workflows & harnesses

Ryan Lavin plans with Claude, then hands engineering prompts to Codex (a "pure engineer"). Says this two-tool split is becoming redundant now that Claude has /goal and dynamic workflows. Built his own Claude↔Codex harness but uses it less — "harnesses are always a short-term solution; the models get better and the harnesses become outdated." Now a desktop-UI maximalist (moved off the CLI).

Nathan runs an agent harness (Open Claw, considering Hermes) on a home Mac Mini, operated entirely through Telegram. Reluctant to migrate harnesses ("don't want to break anything"). Notes Hermes is lighter-weight with better self-improvement / self-updating skills.

Ryan Crosby spins up Claude instances inside VS Code on Windows; still likes seeing the generated code, specs, and file changes live.

Self-improving agents: multiple people now let agents write/find their own skills and update their own memory files in the background. Ryan Lavin's tip: when the agent errs, tell it "add this to your memory, don't let it happen again."

3Testing — the new core of "coding"

Ryan Lavin's biggest time sink is end-to-end testing, not writing code. The architecture is shifting so that everything is an API, enabling exhaustive E2E runs.

  • Fable is praised as an "amazing adversarial actor" — stress-tests systems far harder than Opus, and runs ~20 agents in parallel. E2E suites that took ~7 hours now finish in ~1 hour (but burn most of his tokens).
  • He writes long, realistic scenario tests (sign up → cancel → re-sign up → change details → message physician 150×) and increasingly just lets the agent generate scenarios, glancing over 2–3 sentence summaries. Ryan Crosby works similarly.

Real example of why this matters: an invite/sign-up edge case (user gets the email, never sets a password, exits, returns → locked out) slipped through Ryan Crosby's investor dashboard — exactly the kind of bug thorough E2E would catch.

"If it runs, it runs. It doesn't write code like a human — it's totally alien to me."

4Projects shared

Ryan Crosby — CrumbLabz

Custom software for small businesses. Default stack: simple front end, Firebase (DB + auth), Resend (email), Vercel (hosting, auto-deploy from GitHub, branch/preview management). Pitch: build a free working demo, turn it around in 24–48 hrs, then ~$500/mo. Active/pipeline clients: HOA, real-estate investment firm, aluminum manufacturer, healthcare company, law firm, manufacturers, two development companies, interior-design group.

Ryan Lavin — Telehealth startup

A small Hims competitor, built with his dad: full software stack from intake questionnaire → physician approval → pharmacy. Building a B2B API integration; landing-page builds have gotten dramatically faster. Says the work that once needed 5–10 engineers is mostly just him.

Nathan — Ranch app

Cattle-ranch management (registered Angus) run through Telegram. Parses casual ranch-hand messages ("gave this one 3.5mg") to auto-update records; answers natural-language queries against the DB; generates invoices and Facebook Marketplace sale packages. Uses GPT 5.5; data stored in JSON files. Also doing notably successful SEO (#1 on Google + AI search for "Mississippi Red Angus" within ~2 weeks).

5Industry & future tangents

  • Dictation: strong endorsement of Whisper Flow (mapped to the Fn key). "You type at 60 wpm, you speak at 150–180" — more context, fewer RSI / carpal-tunnel issues.
  • HTML over Markdown for all generated documents — better visuals, diagrams, and reading experience.
  • Anthropic Partner Program: Ryan Crosby exploring it for CrumbLabz — extra tokens, cheaper/earlier model access for those who demonstrably spread Anthropic adoption (referral from Alex Schreiner, who's doing a similar grocery-niche business).
  • Data-moat debate: does proprietary data still matter? Litmus test floated — does Fable outperform Harvey (legal AI)? If yes → data is no moat and SaaS gets commoditized; if no → these companies have a bigger moat than Wall Street thinks. Ryan Crosby: moat exists short-term, breaks down over a decade+.
  • Commoditization analogy: software ~ shoe manufacturing (design files sent off to be "manufactured"). Vibe coding is now mainstream; viral examples like the "82 and oh" sports-draft app were cloned across leagues within a day.
  • Hardware / sensors next: Halter (AI cattle collars, Peter Thiel–backed) as sensors + software extending into the physical world; open-source / open-API sentiment ("API or bust" vs. walled gardens).
  • Reading rec: AI 2027 (a former OpenAI researcher + blogger Scott Alexander) — praised as eerily accurate on AI revenue/trajectory so far.
  • Light investing chatter: cloud computing, hyperscalers, Elon-related, data-center "picks and shovels" (fiber optics, on-site GPU / "home-as-GPU" startups), SpaceX IPO liquidity.

6Key take-aways

On practical usability

  • Pick tools by limits and fit, not hype. Real decisions are driven by usage caps and cost; people switch agents over rate limits as much as raw capability.
  • Don't over-invest in harnesses. They're short-term scaffolding — the models absorb their value (e.g. /goal + dynamic workflows replacing a custom Claude↔Codex pipe). Build light, expect to throw it away.
  • Meet users where they already are. The ranch app's win wasn't the UI — it was Telegram. Users barely touch the app; they message an LLM with DB access and get plain-English answers.
  • Architect everything as an API so agents can exhaustively test and operate against it.
  • Dictation is a real productivity unlock (Whisper Flow): faster input, more context to the model, ergonomic benefit.
  • Generate documents as HTML, not Markdown for readability, visuals, and diagrams.
  • Mind the boring scalability traps. JSON-file storage doesn't scale; move to SQL / Firebase / Neon early — "you don't really need to know how databases work, just the three letters SQL."
  • The recurring trust gap: everyone is comfortable letting agents draft (marketing, outreach, posts) but hesitant to give them the keys to act autonomously — trust is earned through observed reliability over time.

On continuous learning

  • Self-improving agents are becoming the norm — agents that find/write their own skills and edit their own memory files. The "add this to your memory, don't repeat the mistake" loop is a simple, powerful habit.
  • The skill is shifting from writing code to specifying and verifying it. Participants moved from reading every line → reading diffs → reading nothing, trusting "if it runs, it runs" — with the honest caveat that the output is "alien," non-human spaghetti, a real risk worth flagging.
  • Testing is the new core competency. Creativity now lives in inventing adversarial edge-case scenarios; the agent expands and executes them.
  • "All agents are converging on the same product." Differentiation is moving from the model to what you build around it and which niche you serve — multiple people independently landed on "custom software for a vertical, then scale."
  • Stay close to the frontier or fall behind fast. One participant noted falling behind on firsthand Codex/Claude use because he's mobile — a few weeks off is noticeable.
  • New SEO ≈ getting into the training set. Getting your name across the internet so it lands in the next model's pre-training may matter more than classic Google SEO (viewed skeptically as partly "snake oil").
The Treasure

What we're stealing from Voyage I

Six reusable moves any AI builder can apply this week.

  • Make everything an API, then test adversarially. Drive it with a model that thinks like an attacker (Fable), spinning up parallel agents to run realistic multi-step edge cases overnight. The escaped bugs are weird sequences, not happy paths.
  • Meet users in the chat they already live in. A Telegram agent with DB access that parses casual messages into structured records beats a dashboard nobody opens.
  • Run the self-improvement loop. When an agent errs, tell it to write the lesson to its memory file. Let agents find/write their own skills in the background.
  • Ship the boring stack; dodge the boring traps. Firebase + Vercel + Resend turns a custom app around in 24–48h for ~$20/mo. Move off flat JSON to SQL before data outgrows it.
  • The job is now specify + verify, not write. Invent the adversarial scenarios; let the agent expand and execute them — but stay honest that "if it runs, it runs" hides alien, non-human code.
  • Don't marry your harness. It's short-term leverage the next model release will absorb (/goal, dynamic workflows). Build light, niche down, and differentiate on what you build around the model.
