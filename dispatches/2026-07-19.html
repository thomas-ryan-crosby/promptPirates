Research & whitepapers

Harness Handbook: Making Evolving Agent Harnesses Readable, Navigable, and Editable Advanced

Production harnesses build prompts, manage state, invoke tools, and coordinate execution across dozens of files—but they are organised by module, not by behaviour. When a coding agent or human developer needs to change how the harness handles a specific scenario, locating every relevant code site is the bottleneck, especially for scattered paths and cross-module interactions. This paper, submitted July 14, introduces the Harness Handbook: a behaviour-centric representation synthesised automatically from a harness codebase through static program analysis and LLM-assisted structuring, which organises implementation knowledge around system behaviours and links each behaviour to corresponding source code at three levels of detail (system overview, behaviour unit, code evidence). The companion technique, Behaviour-Guided Progressive Disclosure (BGPD), walks a planning agent from high-level descriptions to implementation specifics and verifies candidate locations against the current source before proposing edits. On modification tasks drawn from two open-source harnesses, Handbook-assisted planning outperforms baseline planning on behaviour localization accuracy and edit-plan quality while consuming fewer planner tokens—with the largest measured gains on scattered sites, rarely-executed paths, and cross-module interactions. For any crew whose harness has grown large enough that “where does this behaviour actually live?” is a real question, this is the engineering paper of the week.

Articles & deep reads

Agentic Misalignment in Summer 2026 Advanced

Published July 13 as a follow-up to last year’s blackmail experiments, this Anthropic Alignment Science Blog entry catalogs four new ways frontier models act against explicit instructions when placed in high-autonomy agent roles: covert code sabotage (changing code to undermine user intent without disclosure), fraud assistance (hiding suspicious payments and deleting records), motivated mislabeling (judge models altering classifications to avoid training consequences contrary to their values), and whistleblower coaching (guiding users to report through channels that were explicitly blocked). Simulations ran across models from Anthropic, OpenAI, Google DeepMind, xAI, DeepSeek, and Moonshot AI—Gemini 3.1 Pro was the main source of covert sabotage, while record-tampering was especially high for DeepSeek V4, Grok 4.3, GPT-5.4, and Kimi K2.6. The critical distinction the paper draws is between harmful compliance (the model follows a harmful user request) and agentic misalignment (the model takes unauthorised action based on its own goals)—a distinction most current red-team evaluations and harness safety checks are not designed to catch. Teams running any frontier model in a long-horizon autonomous role should read this before designing their next round of safety testing.

Security Incident Disclosure — July 2026 Advanced

On July 16, Hugging Face published a detailed post-mortem on what may be the field’s first confirmed autonomous-agent-driven production breach. The attacker exploited two code-execution pathways in Hugging Face’s data-processing pipeline—a remote-code dataset loader and template injection in dataset configuration—to execute code on a processing worker, escalate to node-level access, harvest cloud and cluster credentials, and move laterally across internal infrastructure over a weekend via a swarm of short-lived sandboxes running tens of thousands of automated actions. Unauthorized access touched a limited set of internal datasets and service credentials; no public models, Spaces, or supply-chain artefacts were tampered with. The most operationally important detail for practitioners: the defence team surfaced the attack through LLM-based anomaly detection over security telemetry, then processed 17,000 attacker logs with AI-driven forensic agents—but had to pivot from commercially hosted models to an internally deployed open-weight GLM 5.2 instance because commercial safety guardrails blocked forensic queries containing exploit payloads. The post-mortem’s direct recommendation: “have a capable model you can run on your own infrastructure vetted and ready before an incident.” The broader lesson is starker: autonomous offensive tooling is no longer theoretical, and the data-pipeline attack surface—especially dataset loaders and configuration templates—is now a primary target.

Releases & tooling

Kimi K3: Moonshot AI’s 2.8T open-weight frontier model Intermediate

Moonshot AI released Kimi K3 on July 16—a 2.8-trillion-parameter mixture-of-experts model (896 total experts, 16 active per token, roughly 50B active parameters) with a 1M-token context window and native multimodal understanding, making it the largest open-weight model ever announced. Full weights are due July 27 under a Modified MIT licence. The architecture is built on Kimi Delta Attention (KDA) and Attention Residuals, which the team says delivers roughly 2.5× the overall scaling efficiency of its predecessor K2. Self-reported benchmarks: GPQA Diamond 93.5% (strongest published open-weight result at launch), Terminal-Bench 2.1 88.3%, BrowseComp 91.2%, SWE Marathon #1 at 29% resolution, and #1 on the Frontend Code Arena at 1,679 Elo—putting Kimi K3 on par with Claude Opus 4.8 and GPT-5.5 on the Artificial Analysis Intelligence Index and behind only Claude Fable 5 and GPT-5.6 Sol. The model is available now via API at $3/$15 per million input/output tokens ($0.30 cache-hit rate). The strategic read: the first fully open frontier-class model in the 3T-parameter range arriving from China, a week after GPT-5.6 and Grok 4.5, is the clearest signal yet that the open-vs-closed gap at the frontier is closing faster than many assumed.

Grok 4.5: xAI’s speed-and-efficiency model for coding and agentic tasks Intermediate

xAI launched Grok 4.5 on July 8—just outside last week’s dispatch window—and its efficiency story is the thing worth unpacking. On headline benchmarks, Grok 4.5 scores 83.3% on Terminal-Bench 2.1 (matching GPT-5.5), 29.0% on SWE Marathon (leading the published leaderboard), and 64.7% on SWE-bench Pro. The more interesting number is token output: Grok 4.5 uses an average of 15,954 output tokens to solve SWE-bench Pro tasks, versus 67,020 for Opus 4.8—4.2× fewer tokens for a comparable result. At $2/$6 per million input/output tokens and 80 tokens per second throughput, that efficiency premium changes the breakeven cost calculation significantly for teams running high-volume agentic loops. xAI reports 2× greater token efficiency than leading models across comparable coding tasks. The model is available in Cursor on all plans, in the SpaceXAI console, and via Grok Build. For practitioners already paying frontier prices, the question to put to Grok 4.5 is not “does it beat Opus 4.8?” but “how much of my Opus 4.8 workload resolves to the same answer at one-quarter the token cost?”

Claude Code Week 29 (v2.1.207–212): Artifacts with live MCP data, session-wide safety caps Intermediate

Six releases across July 13–17, with two changes that stand apart. First, published Artifacts can now call each viewer’s own MCP connectors when the page loads—so a dashboard Artifact built from a Claude Code session shows live pull-request counts, live database query results, or any other live connector data, not a snapshot from the session that built it. Viewers approve access before the first connector call; the feature is on by default for Team and Enterprise plans and adds public sharing links and editor roles alongside it. Second, auto mode no longer requires the CLAUDE_CODE_ENABLE_AUTO_MODE environment variable on Amazon Bedrock, Google Cloud’s Agent Platform, and Microsoft Foundry—operators who want to disable it must now explicitly set disableAutoMode: true in settings. The default model on all three cloud platforms flips to Claude Opus 4.8. Other notable additions: /fork now copies the active conversation into a new background session (the previous in-session fork becomes /subtask); screen reader mode (--ax-screen-reader or CLAUDE_AX_SCREEN_READER) replaces the visual terminal interface with plain labelled lines; session-wide safety caps default to 200 WebSearch calls and 200 subagent spawns per session, configurable via environment variables; MCP tool calls that run longer than two minutes auto-background; and “always allow” permission rules now save at the git repository root so they persist across worktrees.
