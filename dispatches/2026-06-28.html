Research & whitepapers

Are We Ready For An Agent-Native Memory System? Advanced

Wei Zhou and colleagues at Shanghai Jiao Tong University and collaborating institutions deliver the most comprehensive head-to-head of agent memory architectures published to date: twelve representative systems and two reference baselines, evaluated across five benchmark workloads spanning eleven datasets. The paper proposes a decomposition framework that splits memory into four modules—representation and storage, extraction, retrieval and routing, and maintenance—and uses it to explain why aggregate end-to-end benchmarks mislead. No single architecture dominates because the winning design depends entirely on which module is the bottleneck for a given workload. If your agent underperforms on a memory-intensive task, this framework tells you whether to look at the store, the retriever, or the update logic before reaching for a different library.

arXiv 2606.24775 (Zhou et al., SJTU) · June 23, 2026

The 2025 AI Agent Index: Documenting Technical and Safety Features of Deployed Agentic AI Systems Advanced

Presented this week at FAccT ’26 in Montreal, the 2025 AI Agent Index documents the technical capabilities and safety posture of thirty deployed AI agents using public sources and developer correspondence. The finding that concentrates the mind: most safety documentation comes from the largest labs, and Claude Code is the only system in the study for which all eight safety-documentation fields were populated. Eight of the thirty agents carry known security incidents or reported concerns; prompt injection is documented in two of the five browser agents, the category with the widest attack surface. For teams composing harnesses that call third-party agents as subcomponents, this index is the closest thing the field currently has to a safety census—read it before trusting any agent you did not build yourself.

Articles & deep reads

How agents are transforming work Intermediate

OpenAI published its first detailed look at how users are actually deploying Codex agents, and the trajectory numbers are steep. Median research-use is 56 × higher than in November 2025; engineering 27 ×, customer support 32 ×, legal 13 ×. The shift is structural, not just volumetric: by June 2026, 80.6% of users were making requests estimated to correspond to more than thirty minutes of human work, and 70.2% to more than an hour. At the 99th percentile, individuals are generating more than sixty hours of Codex agent turns per day spread across parallel sessions—a working pattern that simply did not exist seven months ago. The piece reads as part report, part market signal: these are the task delegation patterns your clients and competitors are already normalising.

Releases & tooling

Claude Code v2.1.187–195: /rewind, credential guards, OpenTelemetry, auto-mode classifier, CPU optimisations Intermediate

Five releases across four days. The most immediately useful addition is /rewind (v2.1.191, June 24), which lets you resume a conversation from the state just before the most recent /clear—recovering sessions previously lost to accidental wipes. On the security side, v2.1.187 (June 23) ships sandbox.credentials, a setting that blocks sandboxed shell commands from accessing credential files, a meaningful guardrail for any harness running untrusted code inside agent turns. Observability improves in v2.1.193 (June 25) with a claude_code.assistant_response OpenTelemetry log event (redacted by default; enable via OTEL_LOG_ASSISTANT_RESPONSES=1) and a new autoMode.classifyAllShell setting that routes every Bash and PowerShell command through the auto-mode classifier. Also in v2.1.187: remote MCP tool calls that hang longer than five minutes now abort with an error rather than blocking the session indefinitely (override via CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT). CPU use during streaming dropped roughly 37% via 100 ms text-update coalescing; long-session memory growth from the terminal output cache is also reduced.

Claude Code changelog · June 23–26, 2026

Claude API: Sonnet and Haiku rate limits now match Opus across all tiers Intermediate

Effective June 26, 2026, Anthropic equalised the API rate limits for Sonnet, Haiku, and Opus at every usage tier, and consolidated the former five-tier structure into three: Start, Build, and Scale. Most organisations move to a higher tier automatically; no organisation receives lower limits than before, and no action is required. The practical effect for harness builders: the longstanding incentive to substitute a smaller model in high-throughput pipelines purely for rate headroom disappears. If your routing logic currently assigns requests to Haiku for quota reasons rather than quality reasons, that calculus has changed and is worth revisiting.

Anthropic Developer Platform · June 26, 2026

OpenAI Codex Remote reaches general availability: phone control for all paid subscribers Intermediate

Phone-based remote control of live Codex sessions is now available to every paid ChatGPT tier—Plus, Pro, Business, Enterprise, and Education—following the June 25 general-availability launch. The security architecture is worth understanding before deploying it in a team context. Setup requires scanning a QR code generated on the host machine, which binds a specific phone to a specific host workspace. A relay layer maintained by OpenAI then carries only session traffic—prompts, diffs, approval requests, screenshots, and terminal output—between the phone and the host, without opening any inbound port on the host machine. Local projects, credentials, plug-ins, and MCP server configurations stay on the host and never traverse the relay. For teams evaluating whether the remote-access model meets compliance requirements, the data-in-transit scope is materially narrower than a general-purpose remote-desktop alternative.

Benchmarks & evals

NatureBench: Can Coding Agents Match the Published SOTA of Nature-Family Papers? Advanced

FrontisAI’s NatureBench (arXiv 2606.24530, June 23) constructs ninety tasks directly from peer-reviewed Nature-family publications across six scientific domains—biology, chemistry, climate, materials science, medicine, and physics—and evaluates ten frontier agent configurations against them under a strict no-web-search protocol, with each task running in an isolated containerised environment built by the NatureGym pipeline. The headline number: the strongest configuration surpasses the published SOTA on only 17.8% of tasks under the primary criterion (effect size g > 0.1), and SOTA matching reaches at most 47.8%. The diagnostic insight is more useful than the score: trajectory analysis shows that agent success arrives almost entirely through methodological translation—converting the scientific task into a supervised prediction problem the agent already knows how to solve—rather than through genuine procedural or theoretical invention. The NatureGym pipeline and the 150 reference tasks are publicly available so teams can evaluate their own harnesses against the same environment.

Back to the deck Seven items, and the thread running through them is consistent: the gap between what practitioners assume agents can do and what rigorous evaluation reveals they actually do remains wide, and this week sharpened awareness of it on three fronts. The memory systems paper says no library is universally right—match the architecture to the bottleneck. The Agent Index says most deployed systems are flying with minimal safety documentation and several with documented incidents. NatureBench says frontier agents are sophisticated retrieval engines, not scientists. The platform news says the tooling continues to mature around them: rate limits ease, observability improves, remote access narrows its security surface. The ship is being built while it sails.
